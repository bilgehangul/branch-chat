### Locked Decisions

**Simplify mode selection flow**
- Clicking "Simplify" in the ActionBubble expands the bubble in-place: the 3-button row morphs to show 4 mode buttons (Simpler, Example, Analogy, Technical) plus a ← back arrow

**Simplification output location**
- Simplified text does NOT replace the original paragraph inline
- A distinct simplification block appears below the paragraph — both original and simplified are visible simultaneously, no toggle needed

**Simplification block visual**
- Tinted background + left border accent bar (blockquote-inspired, styled distinctly from AI prose)
- Header line: `✎ Simplified • [Mode name]  [Try another mode]`
- Clicking "Try another mode" re-opens the mode selection (bubble-style expansion or inline mode picker — Claude's discretion)
- If multiple simplifications are requested (user tries another mode), the block updates to the latest result

**Citation block layout**
- Collapsible; default state: collapsed → single-line pill: `🔎 3 sources found ▼`
- Expanded layout: 3 source rows (clickable title, ↗ opens in new tab, domain badge) + horizontal divider + Gemini-generated note prefixed with 💬 icon
- Toggle (▲/▼) at the block header

**Multiple annotation blocks per paragraph**
- A paragraph can have both a citation block and a simplification block simultaneously
- Stacking order: citation block first, simplification block below (creation order)
- Each block is independent — removing one does not affect the other

**Loading states**
- Skeleton/shimmer placeholder block appears in the block's position below the paragraph while API responds
- Same border/shape as the final block, with animated shimmer lines

**Error states**
- Inline error block in the same position as the placeholder on API failure
- Content: `⚠ Couldn't load sources — [Retry]` (or equivalent for Simplify)
- Muted red/orange border treatment
- Clicking Retry re-triggers the same API call

**Re-selectability**
- Both citation blocks and simplification blocks must not interfere with `data-paragraph-id` on the original paragraph element
- The original paragraph remains selectable for Go Deeper / Find Sources / Simplify after annotation

- "Try another mode" interaction detail (inline re-expand vs mini popover)
- Tooltip copy for each mode
- Exact dismiss/remove behavior for annotation blocks (whether user can close/dismiss a block)

## Summary

Phase 5 wires the two stub ActionBubble buttons (Find Sources, Simplify) to real API calls and renders the results as persistent inline blocks below each annotated paragraph. The Zustand store already has `addAnnotation` implemented. The key architectural work is: (1) extending `ActionBubble` with in-place expanded state for mode selection, (2) building `CitationBlock` and `SimplificationBlock` components, (3) wiring them into `MessageBlock` to render from `message.annotations`, and (4) resolving two API shape mismatches discovered in the existing code.

The phase is self-contained on the frontend. The backend `/api/find-sources` and `/api/simplify` routes are fully implemented and protected. No new backend routes are required unless the Gemini citation note needs a dedicated endpoint (see Open Questions — recommended approach: generate the citation note in the find-sources call by calling Gemini from the route, or add a thin `citationNote` field to the existing route response).

**Primary recommendation:** Build `CitationBlock` and `SimplificationBlock` as independent presentational components; manage all async/loading/error state inside `MessageBlock` (or a new `useAnnotations` hook) using local `useState`; persist final results to Zustand via `addAnnotation` / `updateAnnotation`. Do not put loading/error state into Zustand — it is ephemeral UI state.

### Pattern 1: ActionBubble Expanded Mode State

**What:** `ActionBubble` manages an internal `mode: 'default' | 'simplify'` state. In `'simplify'` mode the 3-button row is replaced by 4 mode buttons + ← back. The bubble stays anchored at the same `top`/`left` position (position:fixed).

**When to use:** Anytime a bubble needs multi-step selection without re-positioning.

**Implementation notes:**
- `onFindSources(anchorText: string, paragraphId: string, messageId: string)` added as prop
- `onSimplify(anchorText: string, paragraphId: string, messageId: string, mode: SimplifyMode)` added as prop
- Both props follow the same pattern as `onGoDeeper` — data captured from `bubble` props at call time, NOT from `window.getSelection()`
- All new buttons must call `e.preventDefault()` on `onMouseDown` (established BRANCH-02 pattern)

### Pattern 2: Annotation Rendering in MessageBlock

**What:** `MessageBlock` reads `message.annotations` and renders `CitationBlock` or `SimplificationBlock` components keyed by `annotation.paragraphIndex`. They are injected BELOW the AI bubble `div`, not inside the `MarkdownRenderer` output.

**Critical constraint:** Annotation blocks must be siblings of the message bubble, not children of any element with `data-paragraph-id`. This preserves re-selectability (INLINE-08).

**Structure:**
```tsx
// MessageBlock render
<div data-message-id={message.id}>
  {/* label */}
  <div> {/* bubble */}
    <MarkdownRenderer ... />  {/* data-paragraph-id elements live here */}
  </div>
  {/* annotation blocks — outside the bubble, keyed by annotation.id */}
  {message.annotations.map(ann => ann.type === 'source'
    ? <CitationBlock key={ann.id} annotation={ann} />
    : <SimplificationBlock key={ann.id} annotation={ann} />
  )}
</div>
```

### Pattern 3: Async Loading State (Local, Not Zustand)

**What:** Loading/error/retry state is `useState` inside `MessageBlock` (or a `useAnnotations` hook), NOT in Zustand. Zustand only receives the final `Annotation` record once the API call succeeds.

**Loading flow:**
1. User clicks Find Sources → set `pendingAnnotation: { type: 'source', paragraphId, messageId }` in local state
2. Render `AnnotationShimmer` in the expected block position
3. API call completes → call `addAnnotation(messageId, annotation)` → clear `pendingAnnotation`
4. API call fails → set `errorAnnotation: { ... }` in local state → render error block with Retry button
5. Retry → clear `errorAnnotation`, set `pendingAnnotation`, retry API call

**Why not Zustand for loading:** Loading state is ephemeral; it does not survive re-renders of a different thread; it would require cleanup logic on navigate away. Local state is sufficient and simpler.

### Pattern 4: updateAnnotation Store Action

**What:** `updateAnnotation(messageId, annotationId, patch)` applies a `Partial<Annotation>` to an existing annotation. Required for "Try another mode" — the simplification block updates to the latest result in-place rather than adding a second annotation.

**Implementation** (follows the same immutable update pattern as `updateMessage`):
```ts
updateAnnotation: (messageId: string, annotationId: string, patch: Partial<Annotation>) => {
  set(state => ({
    messages: {
      ...state.messages,
      [messageId]: {
        ...state.messages[messageId]!,
        annotations: state.messages[messageId]!.annotations.map(a =>
          a.id === annotationId ? { ...a, ...patch } : a
        ),
      },
    },
  }));
},
```

### Pattern 5: Shimmer Animation

**What:** Tailwind `animate-pulse` on placeholder divs. Match the border/shape of the final block to prevent layout shift when the real content arrives.

```tsx
// AnnotationShimmer example
<div className="mt-2 rounded-lg border border-slate-200 p-3 animate-pulse">
  <div className="h-3 bg-slate-200 rounded w-1/3 mb-2" />
  <div className="h-3 bg-slate-200 rounded w-full mb-1" />
  <div className="h-3 bg-slate-200 rounded w-4/5" />
</div>
```

### Anti-Patterns to Avoid

- **Mutating `data-paragraph-id` elements:** Do not inject annotation blocks inside the `MarkdownRenderer` output or inside the bubble's prose div. This breaks `data-paragraph-id` and re-selectability.
- **Putting loading/error in Zustand:** Ephemeral UI state (loading, error) must stay in `useState`. Only persisted result state goes to the store.
- **Re-calling `window.getSelection()` at click time:** Selection is already collapsed when the button is clicked. All data (anchorText, paragraphId, messageId) must come from bubble props captured at mouseup time.
- **Accessing `data.text` from simplify response:** Backend returns `{ data: { rewritten } }` not `{ data: { text } }`. See API Shape Mismatches below.

## Critical: API Shape Mismatches

These mismatches exist in the codebase today and MUST be resolved in Phase 5.

### Mismatch 1: `/api/simplify` response field name

**Backend** (`backend/src/routes/simplify.ts` line 31): `res.json({ data: { rewritten }, error: null })`

**Frontend** (`frontend/src/api/simplify.ts` line 7): `Promise<ApiResponse<{ text: string }>>`

**Fix:** Update `simplify.ts` API module to use `{ rewritten: string }` as the response data type, OR update the backend to return `{ data: { text } }`. Recommendation: update the frontend type to match the backend (backend is the source of truth): `Promise<ApiResponse<{ rewritten: string }>>`.

### Mismatch 2: `/api/find-sources` response shape

**Backend** returns: `{ data: { results: SearchResult[] }, error: null }` where `data` is an object wrapping `results`.

**Frontend** declares: `Promise<ApiResponse<SearchResult[]>>` where `data` is expected to be a plain array.

**Fix:** The frontend `searchSources()` call receives `ApiResponse<{ results: SearchResult[] }>` not `ApiResponse<SearchResult[]>`. Update the frontend type: `Promise<ApiResponse<{ results: SearchResult[] }>>` and extract `data.results` when consuming the response.

### Mismatch 3: `SearchResult` vs `SourceResult` field names

**Shared** `SearchResult` (used by backend): `{ title, url, content, score }`

**Frontend** `SourceResult` (used by `Annotation.sources`): `{ title, url, domain, snippet }`

**Fix:** Add a mapping function in the handler that converts `SearchResult` → `SourceResult`:
```ts
function toSourceResult(r: SearchResult): SourceResult {
  return {
    title: r.title,
    url: r.url,
    domain: new URL(r.url).hostname.replace(/^www\./, ''),
    snippet: r.content,
  };
}
```

### Mismatch 4: Citation note endpoint missing

**CONTEXT.md** specifies: citation block shows a "Gemini-generated note" below the sources.

**Current backend:** `/api/find-sources` returns only Tavily search results. There is no endpoint that generates a Gemini-written note about the sources.

**REQUIREMENTS.md** `INLINE-02` states: "A citation note (1–2 sentences, generated by Gemini) is injected below the paragraph".

**Fix options (Claude's discretion):**
- **Option A (recommended):** Extend `/api/find-sources` route to call `aiProvider.simplify()`-style generate after Tavily returns, generating a 1–2 sentence synthesis note. Return it in the response as `{ results, citationNote }`. Single round trip, lower complexity.
- **Option B:** Add a separate `/api/cite-note` endpoint called after sources load. Two round trips, more modular.

Recommended: Option A. The backend route already has `aiProvider` available via `config.ts`.

## Common Pitfalls

### Pitfall 1: Annotation Block Inside `data-paragraph-id` Element
**What goes wrong:** Injecting annotation blocks inside the MarkdownRenderer prose breaks `data-paragraph-id` stability. `rehypeAddParagraphIds` assigns indices sequentially at render time; any DOM child injection inside the prose div that goes through ReactMarkdown will re-number paragraphs.
**Why it happens:** Trying to render "below the paragraph" by appending to the paragraph's children.
**How to avoid:** Render annotation blocks outside the `MarkdownRenderer`/bubble div entirely, as siblings of the bubble in `MessageBlock`.
**Warning signs:** Go Deeper stops creating branches at the correct paragraph; pill positions jump after Find Sources is clicked.

### Pitfall 2: Selection Collapsed When Button Clicked
**What goes wrong:** `window.getSelection()` returns empty string inside `onFindSources`/`onSimplify` click handlers.
**Why it happens:** Clicking a button steals focus, collapsing the browser's text selection.
**How to avoid:** All data (anchorText, paragraphId, messageId) must be captured into bubble state at `mouseup` time and passed as props to ActionBubble. Handlers read from props, not from `window.getSelection()`. This is already the established pattern for `onGoDeeper`.
**Warning signs:** anchorText is empty string when the API call is made.

### Pitfall 3: Multiple Simplification Annotations Per Paragraph
**What goes wrong:** "Try another mode" adds a second `Annotation` record for the same paragraph, resulting in two simplification blocks.
**Why it happens:** Calling `addAnnotation` again instead of `updateAnnotation`.
**How to avoid:** Check `message.annotations` for an existing simplification annotation with the same `paragraphIndex` before creating a new one. If found, call `updateAnnotation(messageId, existingId, { replacementText: newText })`. Use `addAnnotation` only for the first simplification of a paragraph.
**Warning signs:** Two "✎ Simplified" blocks appear below the same paragraph.

### Pitfall 4: ActionBubble Mode State Not Reset After Dismiss
**What goes wrong:** The bubble re-opens in "simplify mode" (showing 4 mode buttons) for the next selection because internal state was not cleared.
**Why it happens:** `clearBubble()` in `ThreadView` clears the parent's `bubble` state, but ActionBubble's internal `mode` state is not reset because the component re-mounts with the same identity.
**How to avoid:** Add `useEffect(() => { setMode('default'); }, [bubble.paragraphId])` inside ActionBubble to reset mode when the selection changes. Or reset in the `onDismiss` callback.
**Warning signs:** Bubble opens in simplify-mode for paragraphs that haven't been interacted with.

### Pitfall 5: `updateAnnotation` Produces Stale Closure
**What goes wrong:** "Try another mode" updates the wrong annotation because the annotationId captured in the closure is stale.
**Why it happens:** The handler captures `annotationId` at the time of the first simplification click, but by the time it's used in the "Try another mode" handler the store may have changed.
**How to avoid:** Read the current annotationId from `message.annotations` at call time using `useSessionStore.getState()` or derive it from the component's props.
